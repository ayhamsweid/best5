# تقرير التدقيق الأمني — Best5

التاريخ: 2026-06-29
النطاق: كود المشروع، إعدادات Docker، Git، الاعتماديات، والخدمات المحلية على `localhost:3000/4000/5432`.

## الخلاصة التنفيذية

الحالة الحالية **حرجة وغير مناسبة للنشر**. أخطر سلسلة هجوم مؤكدة هي:

1. المستودع عام ويحتوي أسرار التشغيل ونسخة PostgreSQL خام كاملة.
2. سر JWT المنشور يسمح بإنشاء توكن مدير دون معرفة كلمة مرور.
3. تم إثبات ذلك محلياً: `/api/users` أعاد `401` بلا توكن، ثم `200` بتوكن مدير مزوّر وكشف حسابَي الإدارة.
4. قاعدة البيانات وواجهة API منشورتان على جميع واجهات الجهاز، ومستخدم قاعدة البيانات يملك `SUPERUSER` و`CREATEDB` و`CREATEROLE`.

يلزم احتواء فوري وتدوير الأسرار، وليس مجرد تعديل الكود.

## حالة المعالجة المحلية بعد التدقيق

تم تنفيذ ما يلي محلياً بتاريخ 2026-06-29:

- تدوير أسرار access/refresh وكلمتي مرور دورَي PostgreSQL دون طباعتها.
- إلغاء تتبع `server/.env` و`db_raw_data_backup/` مع إبقائهما محلياً، وإضافتهما إلى `.gitignore`.
- إزالة كلمات مرور وحسابات المديرين الثابتة من seed.
- إغلاق نشر 4000 و5432 من Docker، وربط الويب بـ`127.0.0.1:3000` فقط.
- فصل دور migrations `best5_user` عن دور التشغيل المحدود `best5_app`.
- منع JWT القديم والمزوّر، وربط التوكن بالمستخدم الفعّال ودوره الحالي، وتقوية refresh.
- إصلاح XSS وJSON-LD والروابط وiframes، وإضافة CSP ورؤوس حماية.
- تقييد رفع الصور والتحقق من بصمتها الثنائية، وتعطيل DB tools.
- إصلاح تجاوز rate-limit وIDOR الخاص بالإشعارات.
- تشغيل API وrenderer كمستخدم `node` مع filesystem للقراءة فقط وإسقاط capabilities.
- ترقية الاعتماديات؛ أصبحت نتيجة `npm audit --omit=dev` صفراً في الواجهة والخادم.
- إصلاح تعارض SSR مع rate-limit: الطلبات الداخلية الموثوقة لا تُحسب كعملاء خارجيين، مع cache قصير وإرجاع `503` بدلاً من تحويل أخطاء `429/5xx` إلى صفحات `404`.
- إضافة healthchecks لخدمات API وrenderer والويب، وتفعيل gzip وكاش طويل للأصول ذات الأسماء الموقعة والصور.
- إضافة DTO validation وحدود طول لمسارات analytics وcategories وtags وsettings.
- تقليص حزم الأيقونات الديناميكية من 1654 ملف JavaScript إلى 22 ملفاً عبر قائمة أيقونات محددة.
- التحقق من جميع روابط sitemap: أصبحت الروابط المدرجة النهائية تعيد `200` بعد حذف صفحات redirect وnoindex منها.
- إصلاح سبع صور مكسورة داخل المقالات، ومزامنة مكتبة الوسائط إلى 261 ملفاً، وتصحيح رابطَي مقال الحلقوم وفنادق تقسيم مع تحويلات `301`.
- إزالة تاريخ March 2026 الثابت من 31 مقالاً واستخدام `updated_at`، واستكمال حقول SEO المفقودة والوسوم الناقصة.

ما يزال الخطر العام **حرجاً** إلى أن تُنفذ الخطوات الخارجية التالية:

- جعل مستودع GitHub خاصاً وتنظيف كامل تاريخ Git ثم force-push منسق.
- تغيير كلمات مرور المديرين يدوياً ومراجعة السجلات.
- ضبط `COOKIE_SECURE=true` في بيئة HTTPS الفعلية.
- فحص خدمة `PostgreSQL_For_Odoo` مستقلة على Windows ما زالت تستمع على `0.0.0.0:5432`؛ هذه ليست منفذ حاوية Compose ولا تخص Best5، لذلك لم يتم إيقافها.

## النتائج مرتبة حسب الخطورة

### C-01 — تسريب عام لقاعدة البيانات والأسرار

**الخطورة: حرجة**

- مستودع GitHub عام، و`origin/main` مطابق لـHEAD المحلي.
- `server/.env` متتبع في Git رغم وجوده في `.gitignore`.
- `db_raw_data_backup/` متتبع: 1341 ملفاً بحجم يقارب 72.6MB.
- الفحص الثنائي أكد وجود بيانات شبيهة بعناوين البريد داخل 10 ملفات وbcrypt hashes داخل 3 ملفات.
- `docker-compose.yml` يحتوي DATABASE_URL وسرّي JWT صراحة.
- `server/prisma/seed.js` يحتوي حسابَي مدير وكلمات مرور ثابتة؛ كلمة المرور الضعيفة لم تعد تعمل على القاعدة الحالية، لكنها ستُعاد عند تشغيل seed.

**الأثر:** استخراج البيانات والمحتوى وعناوين IP والبريد وpassword hashes، تزوير الجلسات، والوصول إلى قاعدة البيانات أو أي بيئة أعادت استخدام الأسرار.

**الإصلاح الفوري:**

1. اجعل المستودع خاصاً فوراً أثناء الاحتواء.
2. دوّر JWT access/refresh، وكلمة مرور PostgreSQL، وكل قيمة داخل `.env`، وكلمات مرور جميع المديرين.
3. اعتبر كل القيم الموجودة في تاريخ Git مكشوفة حتى لو لم تعد مستخدمة.
4. احذف `server/.env` و`db_raw_data_backup/` من Git ومن كامل التاريخ باستخدام `git filter-repo` أو BFG، ثم force-push منسق.
5. خزّن النسخ الاحتياطية مشفرة خارج Git، واستخدم Docker secrets أو secret manager.
6. أضف `db_raw_data_backup/` ونسخ قواعد البيانات إلى `.gitignore`، وفعّل secret scanning/pre-commit.

### C-02 — تجاوز كامل للمصادقة عبر تزوير JWT

**الخطورة: حرجة — مثبت عملياً**

- سر access ثابت ومنشور في `docker-compose.yml`.
- الحارس يثق مباشرة بقيم `id` و`role` داخل JWT.
- تم إنشاء JWT بالهوية `ADMIN` ونجح طلب `/api/users` بحالة `200`.
- endpoint التحديث قبل أيضاً refresh JWT لمستخدم غير موجود، أصدر access cookie، ثم سمح بالوصول الإداري بحالة `200`.

**الأثر:** صلاحيات مدير كاملة: المستخدمون، المقالات، الإعدادات، السجلات، الرفع، وأدوات قاعدة البيانات.

**الإصلاح:**

- أنشئ أسراراً عشوائية قوية (32 بايت على الأقل) خارج المستودع ودوّرها فوراً.
- أضف `iss` و`aud` و`jti` و`token_type`، وافصل مفاتيح access وrefresh فعلياً.
- في كل refresh، اقرأ المستخدم من قاعدة البيانات وتحقق من وجوده و`is_active` ودوره الحالي.
- استخدم refresh-token rotation مع تخزين hash وإبطال العائلة عند إعادة الاستخدام.
- لا تسمح بقيم fallback مثل `dev-access` و`dev-refresh` في production؛ أوقف الإقلاع إذا غاب السر.

### C-03 — PostgreSQL مكشوف ومستخدم التطبيق Superuser

**الخطورة: حرجة**

- المنفذ `5432:5432` مربوط على `0.0.0.0` و`[::]`.
- الاتصال داخل شبكة Compose بالاعتماد المنشور نجح.
- الدور `best5_user` يملك `SUPERUSER`, `CREATEDB`, و`CREATEROLE`.
- API أيضاً مكشوف مباشرة عبر `4000:4000` متجاوزاً Nginx.

**الأثر:** السيطرة الكاملة على PostgreSQL وتجاوز مبدأ أقل صلاحية، مع توسيع أثر أي SQL injection أو تسريب اعتماد.

**الإصلاح:**

- احذف نشر منفذ PostgreSQL؛ استخدم `expose`/الشبكة الداخلية فقط، أو اربطه بـ`127.0.0.1` للتطوير حصراً.
- احذف نشر `4000` في الإنتاج واجعل الوصول إلى API عبر Nginx فقط.
- أنشئ دور تطبيق غير superuser ولا يملك إنشاء قواعد أو أدوار، بصلاحيات على schema المطلوب فقط.
- استخدم دوراً منفصلاً ومؤقتاً للمهاجرات.

### H-01 — Stored XSS في محتوى المقالات والروابط

**الخطورة: عالية**

- `content_ar/content_en` يُمرّر مباشرة إلى `dangerouslySetInnerHTML`.
- `fallbackHtml` يُمرّر بالطريقة نفسها.
- JSON-LD في الواجهة يستخدم `JSON.stringify` داخل `<script>` دون تحويل `<` إلى `\u003c`.
- روابط CTA وiframe تقبل URL غير مقيد؛ يمكن إدخال schemes أو origins غير موثوقة.
- كاتب المحتوى يستطيع تعديل مقال منشور مع إغفال `status`، فيبقى منشوراً ويصل payload للزوار.

**الأثر:** تنفيذ JavaScript في أصل الموقع، والتصرف بجلسة مدير يزور الصفحة، وقراءة CSRF cookie ثم تنفيذ عمليات إدارية بنفس الأصل.

**الإصلاح:**

- عقّم HTML بقائمة سماح موثوقة مثل DOMPurify/isomorphic-dompurify قبل العرض والتخزين.
- الأفضل إلغاء HTML الخام وتمثيل المحتوى بعناصر structured blocks فقط.
- استخدم serializer آمن داخل `<script>` يحوّل `<`, `>`, `&`, U+2028 وU+2029.
- اسمح فقط بـ`https:` وبقائمة hosts للروابط وiframes، وارفض `javascript:`, `data:` و`file:`.
- أضف CSP صارمة.

### H-02 — رفع ملفات غير مقيد في غلاف المقال

**الخطورة: عالية**

المسار `POST /api/posts/:id/cover` لا يضع حد حجم، ولا `fileFilter`، ويحتفظ بامتداد اسم الملف ثم يخدمه من نفس origin. مسار الرفع الآخر أفضل، لكنه يثق بـMIME والامتداد ولا يفحص magic bytes.

**الأثر:** رفع HTML أو ملفات ضخمة، stored phishing/XSS عند فتحها، واستهلاك القرص.

**الإصلاح:**

- وحّد الرفع في خدمة واحدة بحدود حجم وعدد.
- افحص magic bytes، ثم أعد ترميز الصور server-side واحذف metadata.
- ولّد امتداداً من النوع المكتشف، لا من اسم المستخدم.
- اخدم الملفات من origin منفصل بلا cookies مع `Content-Disposition` و`nosniff`.

### H-03 — إدارة جلسات وتحديث غير آمنين

**الخطورة: عالية**

- refresh يقبل التوكن من body أو cookie، ولا يتحقق من `token_type`.
- لا يتحقق من أن المستخدم موجود أو فعال، ولا من دوره الحالي.
- حذف/تعطيل المستخدم أو تغيير دوره لا يبطل التوكنات الحالية.
- لا توجد rotation/revocation، وrefresh صالح 7 أيام.

**الإصلاح:** طبّق التحقق والتدوير والإبطال المذكور في C-02، واحصر refresh في HttpOnly cookie.

### H-04 — اعتماديات production مع ثغرات معروفة

**الخطورة: عالية**

نتيجة `npm audit --omit=dev`:

- Backend: 16 تنبيهاً؛ 7 عالية و9 متوسطة.
- Frontend: تنبيهان متوسطان في `react-router-dom@6.30.3`/`react-router` (open redirect advisory).
- أبرز backend packages المتأثرة: Multer، Nest/Express، lodash، tar، minimatch وpath-to-regexp.

**الإصلاح:** حدّث React Router إلى إصدار مصحح، وخطط لترقية NestJS/Express/Multer مع regression tests. شغّل audit وcontainer scanning في CI وفشل البناء على high/critical القابلة للاستغلال.

### M-01 — تجاوز rate-limit عبر X-Forwarded-For

**الخطورة: متوسطة**

`app.set('trust proxy', 1)` مع نشر API مباشرة يجعل العميل قادراً على إرسال `X-Forwarded-For`. تغيير العنوان أعاد عداد login من جديد عملياً.

**الإصلاح:** لا تنشر API مباشرة، واضبط trust proxy على شبكة/عنوان reverse proxy الموثوق، واحذف/أعد كتابة X-Forwarded-For عند الحافة. أضف rate-limit موزعاً حسب الحساب وIP مع backoff.

### M-02 — CSP ورؤوس حماية الويب والكوكيز

**الخطورة: متوسطة**

- CSP معطلة صراحة في Helmet.
- استجابة الويب عبر Nginx لا تحتوي CSP/HSTS/X-Frame-Options/X-Content-Type-Options.
- `COOKIE_SECURE=false` في Compose.
- Nginx يكشف إصداره في `Server`.

**الإصلاح:** فعّل Secure في HTTPS، وCSP nonce-based، وHSTS بعد ضمان HTTPS، و`nosniff`, `frame-ancestors`, `Referrer-Policy`, `Permissions-Policy`، و`server_tokens off`.

### M-03 — الحاويات تعمل root دون تقليل صلاحيات

**الخطورة: متوسطة**

API وrenderer وweb بلا `USER` مخصص، وبلا read-only filesystem أو إسقاط capabilities.

**الإصلاح:** شغّل Node كمستخدم غير root، واستخدم مستخدم Nginx غير مميز، `read_only`, `no-new-privileges`, `cap_drop: [ALL]` مع إضافة الضروري فقط، وtmpfs لـ`/tmp`.

### M-04 — أدوات استعادة قاعدة البيانات مفعلة وغير محدودة

**الخطورة: متوسطة**

- `ENABLE_DB_TOOLS=1` دائماً.
- restore upload بلا حد حجم أو فحص نوع.
- الميزة تعتمد `pg_dump`/`psql` لكن صورة API لا تثبت PostgreSQL client، لذا تبدو معطلة تشغيلياً حالياً.

**الإصلاح:** عطّلها افتراضياً، وافصلها في job إداري خارج web process مع مصادقة إضافية وتدقيق، وحد حجم، وتحقق، ونسخة احتياطية تلقائية قبل الاستعادة.

### M-05 — endpoint SEO مستثنى من rate-limit

**الخطورة: متوسطة**

`/api/seo/render` مستثنى من المحدد العام وينفذ استعلامات قاعدة بيانات وSSR لكل طلب، ما يسمح بإجهاد التطبيق.

**الإصلاح:** حدّد المعدل، أضف cache، وحد طول/شكل `x-original-uri`.

### L-01 — IDOR في تعليم الإشعار كمقروء

**الخطورة: منخفضة**

`markRead(userId, id)` يتجاهل `userId` ويحدّث حسب `id` فقط. أي مستخدم مصادق يمكنه تعليم إشعار مستخدم آخر كمقروء إن عرف UUID.

**الإصلاح:** استخدم `updateMany({ where: { id, user_id: userId } })` وتحقق أن `count === 1`.

## نقاط الحماية التي اجتازت الفحص

- مسارات الإدارة ترفض الطلب بلا JWT.
- CORS لم يمنح `Access-Control-Allow-Origin` لأصل غريب.
- CSRF double-submit موجود للطلبات غير الآمنة ذات cookies.
- Swagger غير متاح في production.
- محاولات path traversal لم تقرأ ملفات النظام أو `.env`.
- Prisma مستخدم دون raw SQL ظاهر؛ لم يُثبت SQL injection.
- قائمة المستخدمين لا تعيد `password_hash`.
- build الواجهة وSSR والخادم نجح.
